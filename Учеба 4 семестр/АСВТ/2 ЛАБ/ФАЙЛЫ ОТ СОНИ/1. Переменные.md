#### Константы (`#define`)

```c
#define TIMER_TICKS_HALF_PERIOD  200
```

`#define` — это **не переменная**, это просто текстовая замена. Перед компиляцией компилятор везде где видит `TIMER_TICKS_HALF_PERIOD` подставляет `200`. Памяти не занимает.

**Почему 200?** Таймер срабатывает ~806 раз в секунду. Нам нужно менять состояние каждые 0.25 секунды:

```
806 × 0.25 ≈ 200 тиков
```

```c
#define EEPROM_MODE_ADDR  ((uint8_t *)0x00)
```

Адрес в EEPROM где хранится номер режима. `0x00` — самый первый байт EEPROM. `(uint8_t *)` — приведение к типу указателя, этого требует функция `eeprom_read_byte`.

#### Почему переменные `volatile`?

```c
volatile uint8_t mode  = 1;
```

Ключевое слово `volatile` говорит компилятору:

> "Эта переменная может измениться **в любой момент** — не оптимизируй работу с ней"

Без `volatile` компилятор мог бы закэшировать значение в регистре и не перечитывать из памяти. Но наши переменные меняются **в ISR** — прямо во время выполнения основного кода. Поэтому `volatile` обязателен для всех переменных которые используются и в ISR и в main.

#### Переменные

```c
volatile uint8_t mode = 1;
```

- `uint8_t` — беззнаковое 8-битное целое (0-255)
- Хранит **текущий режим**: 1, 2 или 3
- Начальное значение `1` — но потом сразу перезаписывается из EEPROM в main
- Меняется в `ISR(INT0_vect)` и `ISR(INT1_vect)`

```c
volatile uint8_t state = 0;
```

- Хранит **текущее состояние** гирлянды: только 0 или 1
- `0` → первое состояние (например 0xFF)
- `1` → второе состояние (например 0x00)
- Меняется в `ISR(TIMER0_COMP_vect)` через `state ^= 1`
- Отображается на **PD4**


```c
volatile uint8_t y = 0x55;
```

- Хранит **введённое пользователем число** для режима 3
- Начальное значение `0x55 = 01010101` — задано в условии варианта
- Меняется только через кнопки PORTC при зажатом PD7
- В режиме 3:

```
Состояние 0 → выводится y
Состояние 1 → выводится y ^ 0x80
```


```c
volatile uint16_t tick_count = 0;
```

- `uint16_t` — **16-битное** целое (0-65535)
- Считает тики таймера T0
- Почему не `uint8_t`? `uint8_t` максимум 255 — нам нужно считать до 200, хватило бы. Но `uint16_t` взят с запасом на случай если `TIMER_TICKS_HALF_PERIOD` увеличится
- Сбрасывается в 0 когда достигает 200


```c
volatile uint8_t need_update = 0;
```

- **Флаг** (0 или 1) — нужно ли обновить PORTD
- Устанавливается в `1` в ISR кнопок когда сменился режим
- Main видит флаг → обновляет PORTD → сбрасывает флаг в `0`
- Зачем флаг а не сразу обновить в ISR? ISR должна быть короткой, а обновление PORTD лучше делать в main

```c
volatile uint8_t need_eeprom_save = 0;
```

- **Флаг** (0 или 1) — нужно ли записать режим в EEPROM
- Устанавливается в `1` в ISR кнопок
- Main видит флаг → пишет в EEPROM → сбрасывает в `0`
- Почему нельзя писать в EEPROM прямо из ISR? Запись в EEPROM **медленная** (~3.4 мс) — ISR завис бы на это время и пропустил бы другие прерывания
