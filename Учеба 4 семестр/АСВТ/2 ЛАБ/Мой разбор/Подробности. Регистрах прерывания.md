### MCUCR — выбираем когда срабатывает прерывание

MCUCR (MCU Control Register) — регистр настройки микроконтроллера. Нас интересуют только 4 младших бита: ISC01, ISC00 управляют INT0, ISC11, ISC10 управляют INT1.

Из методички — таблица условий срабатывания:

|ISCx1|ISCx0|Условие|
|---|---|---|
|0|0|низкий уровень (кнопка держится)|
|0|1|любое изменение|
|1|0|фронт 1→0 (нажатие)|
|1|1|фронт 0→1 (отпускание)|

В нашем коде оба бита = 1, значит оба прерывания срабатывают по фронту 0→1, то есть когда кнопку отпускают.

### Почему три шага, а не один?

Представь три уровня разрешения:

MCUCR говорит "INT0 срабатывает вот по такому условию". GICR говорит "INT0 вообще разрешён". sei() говорит "все прерывания разрешены глобально". Все три должны быть выполнены — иначе ничего не сработает.

### need_update и need_eeprom_save — зачем эти флаги?

Вот проблема: ISR (обработчик прерывания) должен работать максимально быстро. Пока мы в ISR — другие прерывания заблокированы. Если там делать тяжёлые операции вроде записи в EEPROM — мы надолго "заморозим" систему.

Поэтому используют паттерн "отложенного выполнения":

ISR только ставит флажок — "эй, main(), сделай вот это при случае". А main() в своём бесконечном цикле смотрит на эти флажки и выполняет тяжёлую работу сам.

```
// В ISR — только быстро:
need_eeprom_save = 1;
need_update = 1;

// В main() — уже спокойно:
if (need_update) {
    need_update = 0;
    // обновить PORTD...
}
if (need_eeprom_save) {
    need_eeprom_save = 0;
    eeprom_save_mode(); // долгая операция — не страшно
}
```

need_update = 1 означает "нужно обновить светодиоды на PORTD (показать новый режим)". need_eeprom_save = 1 означает "нужно сохранить текущий режим в EEPROM". Оба сбрасываются в 0 сразу после выполнения, чтобы не делать это снова.
