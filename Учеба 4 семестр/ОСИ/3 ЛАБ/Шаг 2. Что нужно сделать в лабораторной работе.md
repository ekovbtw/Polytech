## Ваши 4 обязательные задания

### 1) Вычисления — `expr.cpp`

Нужно по числам `A1 ... AN` и целевому `S` расставить между числами знаки `+` и `-`, чтобы получилось верное равенство, и **посчитать количество решений**.  
Важно:

- рабочие потоки создаются **в начале**;
- нужно перебрать **все** варианты;
- выводить сами варианты не нужно, только их количество;
- время в `time.txt` — только время алгоритма, **без чтения файла и без создания потоков**.

Наиболее вероятная трактовка:  
это задача полного перебора, которую надо распараллелить между потоками.  
Так как в варианте указаны **Windows + Semaphore + Critical Section**, преподаватель, скорее всего, ждет:

- пул рабочих потоков;
- очередь задач или разделение диапазонов перебора;
- ограничение/координацию работы через **Semaphore**;
- защиту общих данных через **CRITICAL_SECTION**.

### 2) QuickSort — `qsort.cpp`

Нужно написать многопоточную быструю сортировку по возрастанию.  
Требования:

- вход: число потоков, размер массива, сам массив;
- выход: число потоков, `N`, отсортированный массив;
- отдельный `time.txt`;
- потоки создаются **один раз в начале**;
- можно использовать **очередь подзадач**;
- маленькие куски массива не надо дробить слишком мелко;
- нельзя допускать взаимных блокировок и активного ожидания.

Наиболее вероятная трактовка:  
это именно **worker pool + очередь подзадач**, где поток берет диапазон массива, сортирует или разбивает его на подзадачи и кладет обратно в очередь.  
Вариант требует **Windows + Mutex + Event**, значит:

- доступ к очереди задач — через **Mutex**;
- пробуждение спящих потоков при появлении задачи — через **Event**.

### 3) MergeSort — `msort.cpp`

Нужно сделать то же самое, что и для QuickSort, но алгоритм — сортировка слиянием. Все требования совпадают с QuickSort.

Наиболее вероятная трактовка:

- Linux;
- пул потоков;
- очередь задач;
- синхронизация через **Mutex + Condition Variable**.

### 4) Философы — `phil.cpp`

Нужно смоделировать задачу обедающих философов.  
Жесткие требования:

- 7 потоков: `main`, 5 философов, 1 управляющий поток;
- решение должно использовать **управляющий процесс/слугу**;
- не должно быть deadlock;
- не должно быть активного ожидания;
- не должно быть бесконечного голодания;
- поведение должно быть **детерминированным**, без случайных чисел;
- формат вывода строго: `TIME:NUM:OLD->NEW`;
- к моменту завершения все философы должны быть в `T`.

Наиболее вероятная трактовка:

- философы не берут вилки напрямую;
- они отправляют запрос управляющему потоку;
- управляющий решает, кому сейчас можно есть;
- в Linux нужно использовать **Semaphore**.
