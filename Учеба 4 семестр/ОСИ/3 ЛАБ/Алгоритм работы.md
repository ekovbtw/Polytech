## QuickSort — многопоточная сортировка (Windows, Mutex + Event)

### 1. Алгоритм QuickSort — объяснение по-человечески

QuickSort работает по принципу «разделяй и властвуй»:

1. Выбираешь один элемент — **опорный (pivot)**
2. Всё, что меньше него — перемещаешь влево, всё что больше — вправо
3. Pivot теперь стоит на своём окончательном месте
4. Рекурсивно делаешь то же самое для левой и правой части

**Пример на `массиве [8, 3, 1, 5, 2]`:**

```
Шаг 1: pivot = 5
  [8, 3, 1, 5, 2]
        ↓
  меньше 5 → [3, 1, 2]  |  [5]  |  больше 5 → [8]

Шаг 2: сортируем [3, 1, 2], pivot = 2
  меньше 2 → [1]  |  [2]  |  больше 2 → [3]

Шаг 3: [1] и [3] — по одному элементу, готово

Итог: [1, 2, 3, 5, 8] ✓
```

### 2. Идея параллельной версии

После того как мы разбили массив на две части — их можно сортировать **независимо**. Это и есть точка параллелизма.

Вместо того чтобы рекурсивно уходить вглубь в одном потоке — мы кладём подзадачи в **общую очередь**, а свободные потоки из пула их разбирают.

```
Очередь: [ весь массив [0..N-1] ]

Поток 1 берёт → разбивает по pivot → кладёт [0..k-1] и [k+1..N-1]
Поток 2 берёт [0..k-1]   → разбивает дальше → кладёт ещё подзадачи
Поток 3 берёт [k+1..N-1] → разбивает дальше → ...
...
Когда кусок ≤ 1000 элементов → сортируем рекурсивно без очереди
```

> **Почему не дробить до 1–2 элементов?** Потому что каждое обращение к очереди — это захват мутекса, накладные расходы. Для маленьких кусков это дороже, чем просто рекурсивно досортировать.

### 3. Объекты синхронизации — простыми словами

По варианту нужно использовать **Mutex** и **Event** (WinAPI).

#### Mutex — «ручка от тетради»

Очередь задач — это общий ресурс. Если несколько потоков одновременно в неё полезут — будет каша. Mutex гарантирует, что в каждый момент **только один поток** работает с очередью.

```
WaitForSingleObject(g_mutex, INFINITE)  ← захватить (взять ручку)
  ... работа с очередью ...
ReleaseMutex(g_mutex)                   ← отпустить (положить ручку)
```

#### Событие — «звонок будильника»

Поток проверил очередь — она пуста. Что делать? Нельзя крутиться в цикле — это **активное ожидание**, методичка его запрещает. Поток должен **заснуть** и проснуться, когда появится задача.

Event с **автосбросом** работает так:

- Поток вызывает → засыпает`WaitForSingleObject(g_event, ...)`
- Другой поток добавил задачу → вызывает → один спящий поток просыпается`SetEvent(g_event)`
- Event автоматически сбрасывается (остальные продолжают спать)

```
Поток добавил задачу → SetEvent()     ← «дзынь»
Спящий поток просыпается              ← «слышу, иду»
```

> **Почему автосброс?** Потому что одна задача — для одного потока. Не нужно будить всех сразу.


### 4. Схема потоков

```
main()
  │
  ├─ читает input.txt
  ├─ создаёт Mutex, Event, Done-event
  ├─ кладёт первую задачу [0..N-1] в очередь
  ├─ запускает N рабочих потоков  ← все сразу, до начала работы
  ├─ замеряет время старта
  ├─ WaitForSingleObject(g_done_event)  ← ждёт завершения всей работы
  ├─ замеряет время конца
  └─ пишет output.txt и time.txt

Рабочий поток (×N):
  loop:
    ├─ захватить мутекс
    ├─ есть задача в очереди?
    │    ДА → взять задачу, ++g_active, отпустить мутекс
    │         → выполнить задачу (см. ниже)
    │         → захватить мутекс, --g_active
    │         → если очередь пуста И g_active == 0 → SetEvent(g_done_event)
    │         → отпустить мутекс
    │    НЕТ → отпустить мутекс
    │         → WaitForSingleObject(g_event, таймаут)
    │         → проверить: если очередь пуста и g_active == 0 → выйти
    └─ повторить

Выполнение задачи [left..right]:
    ├─ если right - left < THRESHOLD (1000) → обычный рекурсивный qsort
    └─ иначе → разбить по pivot → добавить [left..p-1] и [p+1..right] в очередь
               → SetEvent() дважды (разбудить два потока)
```

### 5. Маленький пример прохода

Массив: , 2 потока, порог = 3`[8, 9, 7, 1, 2]`

```
Очередь: [ (0..4) ]
g_active = 0

Поток 1 берёт (0..4), g_active = 1
  pivot = 7 (средний элемент)
  разбиение: [1, 2, 7, 9, 8] → левая (0..1), правая (3..4)
  кладёт в очередь: (0..1), (3..4)
  SetEvent × 2
  g_active = 0 после завершения

Поток 1 берёт (0..1), g_active = 1
  длина = 2 < 3 → рекурсивный qsort → [1, 2]

Поток 2 берёт (3..4), g_active = 2 (одновременно с потоком 1)
  длина = 2 < 3 → рекурсивный qsort → [8, 9]

Оба завершают, g_active = 0, очередь пуста
→ SetEvent(g_done_event)

main просыпается
Результат: [1, 2, 7, 8, 9] ✓
```
